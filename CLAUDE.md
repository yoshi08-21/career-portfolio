# CLAUDE.md

このリポジトリは、業務経験を転職活動用のアピール文へ整理・蓄積するための個人リポジトリ。コード開発用ではない。**特定の会社・プロジェクトに依存しない横断ハブ**として運用し、対象プロジェクトをメンションして経験の整理を依頼する（複数プロジェクトの親ディレクトリに置く。配置・運用は `README.md` / `START-HERE.md`）。

AI は「タスクの関連資料」と「本人の粗いメモ」を元に、経験を**伝わりやすいアピール文**へ整理する役割を担う。

## 整理の基本方針

- 文章は **「課題 → 施策 → 工夫（再現性のある判断・設計）→ 成果」** の流れで構成する。
- 採用担当に響くのは **当事者性**（言われたことの外側で自分が気づき・判断したこと）と **具体性**（定量成果・固有のエピソード）。ここを主役にする。
- **事実の裏付けは関連資料から取る**。資料に無い成果や数値を創作しない。
- 数値・固有名で埋められない箇所は **プレースホルダー化**し、「ここに実数を」と明示する。
- 社名・パートナー名・人物名は**デフォルトで抽象化**する（「あるパートナー」「業務担当(CS)」等）。本人が出してよいと言った場合のみ実名にする。
- 固有名（社内システム名・チケットID・内部テーブル/メソッド名）の扱いは、タスクの進行段階で切り替える（下記「固有名の扱い」）。

## 固有名の扱い（具体↔抽象のライフサイクル）

固有名は「いつ抽象化するか」をタスクの段階で切り替える。**進行中は具体（実名）／完了・レジュメ化時に抽象**が原則。

- **進行中（追記運用中）のタスクは実名で書く**。チケットID・リポジトリ/サービス名・内部テーブル/メソッド名などを具体的に残す。理由: 後続ステップを追記する際の**事実照合とトレーサビリティ**（資料・チケット・コードとの突き合わせ）を優先するため。途中で抽象化すると、追記のたびに実名→抽象の変換が要り、表記ゆれ・対応ミスの温床になる。
- **完了時・レジュメ/ポートフォリオへ転用する時に、まとめて抽象化する**（最後の1パス）。抽象化＝固有名の置換は、完成版を作るタイミングで一度に行う方が整合を取りやすく安全。
- **実名で書くファイルには、その旨と抽象化方針を明示する**。
  - ファイル冒頭に「実名で記述している／レジュメ化時に末尾の用語対応表で一括抽象化する」旨の注記を置く。
  - ファイル末尾に**用語対応表（実名 → 抽象名）**を持たせ、最終抽象化を機械的な置換1パスで済ませられるようにする。
- **一般技術名は抽象化しない**（Go / PHP / Slack / SendGrid / Firebase / GCP 等）。技術が伝わる方が有利なため実名で残す。抽象化対象は**社名・パートナー名・人物名・社内固有名（システム/リポジトリ/チケット/内部実装名）**に限る。
- Git 連携する場合の機密配慮: 実名を含む進行中ドラフトを Git 管理対象外にしたいときは `*.local.md` / `private/` に置く（`.gitignore` 除外済み）。通常運用では `.md` のままでよい。

## 入力チェックリスト（依頼時に確認する）

**必須**
1. 事実の裏付けとなる入力（資料メンションが基本。無ければ git 履歴等。下記「入力ソースの取り方」）
2. 本人の箇条書きメモ = 伝えたい主旨・力点

**あると精度が上がる**
3. 対象プロジェクト／在籍企業の文脈（保存先 `appeals/<会社>/` の決定にも使う）
4. 用途（職務経歴書 / ポートフォリオ / 面接の口頭用）
5. 応募職種・方向性（技術寄り / 業務貢献寄り / マネジメント）
6. 定量成果（before→after の数値）
7. 機密配慮の範囲（伏せる固有名）
8. 「気づき・判断」のエピソード（差別化材料）

必須2つ（入力＋メモ）があれば着手し、足りない項目はプレースホルダーで補って明示する。

## 入力ソースの取り方（柔軟）

このリポジトリは複数プロジェクト（在籍企業）を横断する。プロジェクトごとに資料の整い方が違うため、入力の取り方は柔軟にする。**基本はファイルメンション**だが、無ければ他の手段で素材化する。

- **基本：ファイルメンション** — タスク資料ディレクトリ・PR・チケットを `@<相対パス>` で渡す（例: SUPERNOVA の `sn-personal-notes/tasks/...`）。
- **整理資料が無いプロジェクト：git 履歴から棚卸し** — 対象リポジトリのコミット／PR（`git log` / `gh pr view` 等）から、担当範囲・変更内容・判断を拾って素材化する。
- **本人メモ** — 箇条書きをチャットに貼る／口頭で伝える。
- **将来：Slack 連携** — 接続できたら、関連スレッド・やりとりを素材として参照する。
- **その場で素材を作る** — まとまった資料が無い場合、`sources/<会社>/<テーマ>.md` にタスクメモを新規作成してから整理に進む。

いずれの場合も「**事実の裏付けは入力から取り、無い数値・固有名は創作せずプレースホルダー化**」の原則は変わらない。

## 出力フォーマット

タスクの段階で2つの形式を使い分ける（[[固有名の扱い]]＝具体↔抽象のライフサイクルと対応）。

### A. 進行中ドラフト（`appeal-template.md` 形式）

進行中・追記運用中のタスクはこの形式で蓄積する。1テーマにつき、原則として以下をまとめて出す。

1. **メイン版**（職務経歴書・ポートフォリオ向け）: 背景・課題 / 施策 / 工夫① ② … / 成果
2. **短縮版**（履歴書の自己PR欄・1段落）
3. **さらに伝わりやすくするための補足**（埋めるべき数値・強調点）
4. **面接での深掘り対策**（想定問答・任意）

### B. レジュメ完成版（`resume-template.md` 形式）

タスク完了・レジュメ/ポートフォリオへ転用するときは、上記ドラフトを**この形式へ変換し、同時に固有名を抽象化**する。

- 1プロジェクト＝**詳細版／職務経歴書向け圧縮版**の2バリアント。各バリアントは同じ5見出し（**課題 / 解決策 / 成果 / 関与度 / 使用技術**）。
- タイトルは「〜を実装し、〜により〜を解消」と動詞で締める一文サマリ。
- 課題・成果に**定量**を必ず織り込む（頻度×1件あたり＝合計工数、before→after）。実数が無い箇所はプレースホルダーで明示。
- `関与度`＝当事者性（自ら調査・判断・提案＋担当範囲）、`使用技術`＝カンマ区切り。

成果物は `career-portfolio/appeals/<会社・在籍コンテキスト>/` に `YYYY-MM-<テーマ>.md` で保存する（例: `appeals/supernova/2026-06-delete-order-batch.md`）。会社ディレクトリが未作成なら作業時に切る。

## アピールポイントの取捨選択（レジュメ転記時）

レジュメ・職務経歴書は紙面が限られ、**1プロジェクトにつき載せられるアピールポイントは2〜3点**。各 appeal ファイルに候補を貯めておき（[[出力フォーマット]] A の「アピールポイント候補」表）、転記時に応募先へ合わせて上位を選ぶ。土台は共通・最終選択だけ応募先で出し分ける**ハイブリッド方式**。

### 評価ルーブリック（5軸）

候補ポイントを次の5軸で評価する。4軸は **◎=3／○=2／△=1**、職種タグはラベル付与。

| 軸 | ◎(3) | ○(2) | △(1) |
|---|---|---|---|
| **当事者性** | 言われた外側で自ら論点発見・判断を主導 | 一部自発的な工夫 | 指示範囲の遂行 |
| **定量インパクト** | before→after の明確な数値/業務効果 | 効果はあるが数値が弱い | 定性的 |
| **技術難度・深さ** | 非自明な設計判断・トレードオフ | 標準的な実装上の工夫 | 定型作業 |
| **再現性・汎用性** | 他社でも通用する力 | ある程度移植可能 | 自社固有色が強い |
| **職種タグ** | スコアでなくラベル。`BE` / `SRE/インフラ` / `テックリード` / `PdM/業務` / `フルスタック` / `品質/QA` 等から該当を付与 | | |

- 進行中で数値未確定の項目は **定量を「暫定」** と注記し、確定後に更新する。
- 重み付けの目安: **当事者性を最重視**、次に定量・技術難度。職種タグは応募先での**フィルタ/加点**に使う（厳密計算ではなく判断補助）。

### 選択手順

1. **入力**: 応募職種／求人票（あれば）／1プロジェクトの掲載点数（既定2〜3）。依頼は `templates/selection-request-template.md`。
2. **絞り込み**: 候補表を**職種タグで応募先に合うもの優先**。応募先未定なら職種タグを使わず総合スコアで汎用上位を選ぶ。
3. **ランク付け**: 当事者性を最重視しつつ5軸スコアで序列化。同点は「定量がある」「ストーリーが語れる（面接の深掘りに耐える）」を優先。
4. **出力**: 選んだ2〜3点を**理由（どの軸で効くか・応募先のどの要件に当たるか）付きで提示**し、**落とした点とその理由**も併記する。依頼があればレジュメ完成版（`resume-template.md` 形式）へ整形し抽象化する。
