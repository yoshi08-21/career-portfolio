# career-portfolio

業務で得た経験を、転職活動で使える**アピール文（職務経歴書・ポートフォリオ・面接素材）**へ整理・蓄積するための個人リポジトリ。

`task-management` が「目標 → 行動 → 証跡 → 評価」をつなぐのに対し、このリポジトリは **「経験 → 整理 → アピール」** をつなぐことを目的にする。

## 基本の流れ

```text
タスクの関連資料 + 自分の粗いメモ
        ↓ （AI が整理）
アピール文（課題 → 施策 → 工夫 → 成果）＋ アピールポイント候補を5軸で評価してストック
        ↓ （レジュメ転記時：応募先に合わせAIが上位2〜3点を選定）
レジュメ完成版（課題/解決策/成果/関与度/使用技術）に整形・固有名を抽象化
        ↓
職務経歴書・ポートフォリオ・面接で使う
```

## ディレクトリ構成

| パス | 役割 |
|---|---|
| `sources/` | 経験の素材置き場。タスクの箇条書きメモ・関連資料へのリンクなど、整理前のインプット |
| `appeals/` | 完成・下書きのアピール文（成果物）。1テーマ＝1ファイル |
| `templates/` | 進行中ドラフト用アピール文テンプレ（`appeal-template.md`）・レジュメ完成版テンプレ（`resume-template.md`＝課題/解決策/成果/関与度/使用技術 × 詳細版/圧縮版）・AI への依頼テンプレ（`request-template.md`）・アピールポイント選択依頼テンプレ（`selection-request-template.md`） |
| `resume/` | （任意・必要になったら追加）職務経歴書・履歴書本体 |
| `CLAUDE.md` | このリポジトリで AI に整理を頼むときの方針・入力チェックリスト・出力フォーマット |

## 使い方（1テーマを追加するとき）

別セッションで始めるときの開始ルールは **`START-HERE.md`** にまとめている（何をメンションし、何を共有し、何を頼むか）。

1. `START-HERE.md` のキックオフ文をコピペして、対象タスクの資料と自分のメモを渡す。
2. AI が `appeals/` に下書きを作る（数値・固有名はプレースホルダー）。
3. 実データ（before→after の数値、固有名の伏せ方）を埋めて完成させる。

## 運用メモ

- 在職中の準備のため、**社名・パートナー名・人物名はデフォルトで抽象化**する（「あるパートナー」「業務担当(CS)」等）。
- **進行中タスクは実名（具体）で書き、完了・レジュメ化時にまとめて抽象化**する（最後の1パス）。実名ドラフトには冒頭注記＋末尾の用語対応表を付ける。詳細は `CLAUDE.md`「固有名の扱い（具体↔抽象のライフサイクル）」。
- 機密・個人情報・未公開の下書きは Git 管理対象外にできるよう、`.gitignore` で `*.local.md` / `private/` を除外している。
- GitHub（private リポジトリ）への接続は**次の段階**で行う。現状はローカルのみ。
