# B1 auto-inventory — 実行プレイブック

このファイルに従って棚卸しを行う。あなた（実行するエージェント）はこの手順を上から実行する。
**git は読むだけ。`git add/commit/push` は絶対にしない。** 事実は git/PR から取り、無い情報はプレースホルダーにする。

---

## 0. 入力を確定する

- **対象リポジトリ**: メンションされたパス（例 `@supernova/<repo>`、`@simount/pms/pms-api`）。
- **ステージの判定**:
  - 「<期間> を棚卸し」→ **Stage 1（sweep）**。
  - 「PR #N を深掘り」→ **Stage 2（deep-dive）**。
- **期間**（Stage 1）: 指定が無ければ「直近3ヶ月」を既定にし、本人に確認。台帳があれば「前回棚卸し以降」も提案する。

## 1. 会社ディレクトリを判定する

対象リポジトリの remote を見て出力先の会社を決める（`git -C <repo> remote get-url origin`）。

| remote org | 会社ディレクトリ |
|---|---|
| `supernova` 系 | `supernova` |
| `xincere-inc` | `xincere` |
| `simount` | `simount` |
| `upbond` | `upbond` |

- 表に無い org が出たら**本人に確認**してから `appeals/`・`sources/` に新規ディレクトリを切る（`.gitkeep` 付き）。
- ネストしたリポジトリ（例 `simount/pms/pms-api`）でも会社は org 基準（= `simount`）。

---

## Stage 1: sweep（棚卸し一覧をつくる）

### S1-1. PR を機能単位で抽出
`gh` が使えるなら PR を基準にする（推奨）。

```
gh -R <owner/repo> pr list --author "@me" --state merged \
   --search "merged:>=<開始日>" --json number,title,mergedAt,additions,deletions,files,labels,url \
   --limit 100
```

- `gh` が無い / リモートが GitHub でない場合は `git log` でフォールバック:
  `git -C <repo> log --author=<自分のメール> --since=<開始日> --merges` および通常コミットを機能単位にまとめる。著者メールが不明なら本人に確認。
- レビューのみ参加した PR（自分が author でない）は除外する。

### S1-2. 各 PR を1行サマリ化＋暫定スコア
各 PR について以下を埋める:
- `PR番号 / リポジトリ / タイトル / マージ時期 / 規模（+追加 -削除 行・変更ファイル数）/ 一言サマリ（何をしたか）`
  - ※ 1社で複数リポジトリを扱うため、**リポジトリ列は必須**（台帳は会社単位で1つ）。
- **アピール見込み（暫定 ◎○△）**: git シグナルからの**仮置き**。あくまで目安で、最終判断は本人。
  - 規模が大きい・複数ファイル横断・新機能/設計変更 → 高め（◎）
  - 定型修正・軽微・依存更新のみ → 低め（△）
  - ⚠️ **当事者性・定量インパクトは git からは判定できない**ので、この暫定スコアには含めない（深掘りで判明する）。
- **職種タグの推定**: 変更ファイルのパス/拡張子から `BE / フロント / インフラ / SRE` 等を推定して付す。

### S1-3. 台帳 `sources/<会社>/_inventory.md` に追記
- ファイルが無ければ `templates/inventory.md` を雛形に新規作成。
- 既存行（同じ PR番号）があれば**重複させず更新**。状態列は新規なら `未着手`。
- 行フォーマットは台帳テンプレートに従う。

### S1-4. 機能テーマで束ねる（横断グルーピング）
個々の PR を見るだけでなく、**同じ機能を構成する PR/リポジトリをまとめる**。これが棚卸しの肝。
- 例: BE の新API（#1738/#1760）＋ フロントの繋ぎこみ（#136）＝「メンバーオーダー更新機能」。BE とフロントを跨ぐ仕事は1テーマとして語る方が当事者性・成果が立つ。
- 関連する**既存 appeal**（`appeals/<会社>/`）があれば紐づける（重複作業を防ぎ、素材を合流させる）。
- 束ねた結果を台帳の `## 機能テーマでの束ね` セクションに記録する。

### S1-5. 候補を提示
- 暫定 ◎○ と**束ねたテーマ**を中心に「深掘り推奨」を提示し、**どれ（PR単体 or テーマ束ね）を Stage 2 で深掘りするか**を本人に尋ねる。
- 「全部一覧で見たい」と言われたら台帳の内容を表示する。

---

## Stage 2: deep-dive（候補を深掘りして素材化）

### S2-1. PR の中身を取得
```
gh -R <owner/repo> pr view <N> --json title,body,mergedAt,commits,files,url
gh -R <owner/repo> pr diff <N>
```
- 必要に応じて個別コミットも確認（`git -C <repo> show <sha>`）。PR 説明・関連 issue から背景を拾う。

### S2-2. 自動ドラフトを作る
`templates/source-deepdive.md` を雛形に `sources/<会社>/<テーマ>.md` を作成し、**git から言える事実（What）**を埋める:
- `背景・課題`（PR説明/issueから。不足は `[要確認]`）
- `やったこと(What)`（差分・コミットから事実ベース）
- 出典（PR番号・URL・主なコミット）

### S2-3. インタビュアー役で「埋まらない所」を質問化
`../shared/agents/interviewer.md` の役割で、git に無い箇所を**質問リスト**にする:
- `工夫・判断(Why/How)` / `当事者性メモ` / `成果（定量）` を埋めるための質問。
- 質問は素材ファイルの `## 深掘り質問リスト` に書き込む（本人が後で回答 → 対話で埋めてもよい）。

### S2-4. 検証役で裏付けをチェック
`../shared/agents/verifier.md` の役割で、ドラフトの記述を検証:
- 出典の無い記述・誇張・創作数値を指摘し、数値は `[ここに実数を]` に置換。
- 指摘リストを提示し、必要なら本人確認。

### S2-5. 台帳を更新
- `sources/<会社>/_inventory.md` の該当 PR 行の状態を `深掘り済み` に更新し、素材ファイルへのリンクを記す。

---

## 完了時

- 生成した素材ファイルと台帳の差分を**提示するだけ**で、**コミットはしない**（git 操作は本人の明示指示後）。
- 「この素材を appeal にしますか?」と案内し、appeal 化は B6 等・既存の `appeal-template` 流れに引き継ぐ（B1 の責務はここまで）。
